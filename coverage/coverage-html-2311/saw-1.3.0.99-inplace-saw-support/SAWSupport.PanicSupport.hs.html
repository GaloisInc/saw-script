    9 -- Wrapper around the panic library.
   10 --
   11 -- Each library in SAW defines its own Panic module so that panics in
   12 -- it come out with the "component name" set to the library involved.
   13 --
   14 -- This module contains the shared plumbing to make those modules as
   15 -- small as reasonably possible.
   16 --
   17 -- Panic is for internal/programming errors, not user-facing errors.
   18 -- Ideally users should never see panics, and if they do see a panic
   19 -- they should file a bug report about it.
   20 --
   21 -- Note: any uses of the Haskell "error" function should be converted
   22 -- to use panic instead, provided they are actually panic conditions.
   23 -- Any others should be updated to use proper error handling channels.
   24 --
   25 -- Panic is immediately fatal and may be used in pure ("pure") code
   26 -- when needed.
   27 --
   28 -- The exposed panic interface in SAW takes two arguments:
   29 --    - the "location of the problem" (a single string)
   30 --    - the "problem description" (a list of strings, one per line)
   31 --
   32 -- The infrastructure then adds the "component name" as well as the
   33 -- Haskell module and line (via HasCallStack) and the git commit
   34 -- information, and generates a print of the following form:
   35 --
   36 --    [18:30:30.500] You have encountered a bug in <component name>'s implementation.
   37 --    *** Please create an issue at https://github.com/GaloisInc/saw-script/issues
   38 --
   39 --    %< --------------------------------------------------- 
   40 --      Revision:  abcd1234abcd1234cdef5678cdef5678abcd1234
   41 --      Branch:    master (uncommited files present)
   42 --      Location:  <location of the problem>
   43 --      Message:   <problem description>
   44 --                 <problem description>
   45 --                 <problem description>
   46 --    CallStack (from HasCallStack):
   47 --      panic, called at src/SAWScript/Panic.hs:20:9 in saw-script-1.3-inplace:SAWScript.Panic
   48 --      panic, called at src/SAWCentral/Proof.hs:300:15 in saw-script-1.3-inplace:SAWCentral.Proof
   49 --    %< ---------------------------------------------------
   50 --
   51 -- The arguments to each panic should be written so they make sense in
   52 -- this context.
   53 --
   54 -- Also note that because the description lines are indented after
   55 -- being unlines'd, they should not contain their own newlines.
   56 --
   57 -- It is reasonable to use the function name as the "location of the
   58 -- problem"; however, it's often more helpful to use a short string
   59 -- that describes what SAW was trying to do when things came unstuck.
   60 --
   61 -- Then, the rest of the message should indicate what happened and
   62 -- also include the values of relevant variables. What gets printed
   63 -- here may be all you ever get to try to figure out what broke.
   64 --
   65 -- It's important that the combination of the "location" string and
   66 -- the beginning of the rest of the message should be unique.
   67 --
   68 -- Rationale:
   69 --
   70 -- In an ideal world no user would ever see a panic. However, the
   71 -- reality of software is that sooner or later someone will see every
   72 -- panic in the system. Furthermore, chances are that person will
   73 -- probably be a downstream user, in the field. This will be someone
   74 -- who has little or no familiarity with SAW's internals -- given the
   75 -- nature of SAW they will probably not be a completely clueless end
   76 -- user, but even so internal minutiae won't be helpful.
   77 --
   78 -- Therefore, that user (not the SAW developer) is the primary
   79 -- audience for every panic message. Panic messages should tell the
   80 -- user enough about what happened that they can try to rearrange
   81 -- their stuff to avoid triggering it. They should also tell the user
   82 -- enough that they have a reasonable chance at being able to cut down
   83 -- whatever they're working on to something they can send upstream
   84 -- with a bug report.
   85 --
   86 -- The SAW developer is the _secondary_ audience for every panic
   87 -- message, but in general only via the bug reports that come from the
   88 -- primary audience.
   89 --
   90 -- In most cases the function name where something blew up will not
   91 -- mean much to the user seeing the panic. Hence the recommendation
   92 -- for a short descriptive string instead.
   93 --
   94 -- The function name is also readily found via the module name, line
   95 -- number, and message text. (The reason the message text needs to be
   96 -- unique is that if it's not, it's easy to end up looking at the
   97 -- wrong instance, despite line numbers, and then things become far
   98 -- more confusing.) It's therefore not particularly important to
   99 -- include it in the message if there's a better alternative.
  100 --
  101 -- Because panics in general get reported, not found directly by
  102 -- developers, and are often hard to reproduce locally or on demand,
  103 -- it's routinely necessary to diagnose and fix problems entirely or
  104 -- almost entirely by analysis of the panic message and the code
  105 -- around it. For this reason panics should include as much relevant
  106 -- information as possible without flooding the recipient.
  107 
  108 -- re-export HasCallStack for use by the per-library modules
  109 module SAWSupport.PanicSupport (Panic.HasCallStack, doPanic) where
  110 
  111 import qualified Data.Text as Text
  112 import Data.Text (Text)
  113 import qualified Panic as Panic
  114 
  115 -- | Hook for the panic library's use. The upstream library uses types
  116 --   as hooks to attach the panic metadata. We use a simpler scheme
  117 --   where the metadata is implicit in which panic function you call
  118 --   (at the cost of having a separate panic module in each library)
  119 --   so we only need one of these.
  120 --
  121 --   The argument of the constructor is the component name to use
  122 --   with the panic message.
  123 newtype SAWPanic = SAWPanic Text
  124 
  125 -- | Trigger a panic. This is called by each library's panic function;
  126 --   the first argument should be the component name, and the second
  127 --   and third should be the "location" and "description" to pass
  128 --   through to the upstream panic function.
  129 doPanic :: Panic.HasCallStack => Text -> Text -> [Text] -> a
  130 doPanic component loc descs =
  131   Panic.panic (SAWPanic component) loc' descs'
  132   where
  133      loc' = Text.unpack loc
  134      descs' = map Text.unpack descs
  135 
  136 instance Panic.PanicComponent SAWPanic where
  137   panicComponentName (SAWPanic component) = Text.unpack component
  138   panicComponentIssues _ = "https://github.com/GaloisInc/saw-script/issues"
  139 
  140   {-# Noinline panicComponentRevision #-}
  141   panicComponentRevision = $Panic.useGitRevision
