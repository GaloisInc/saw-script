 1990 {- Note [Error recovery]
 1991 ~~~~~~~~~~~~~~~~~~~~~~~~
 1992 When there is no applicable action for the current lookahead token `tk`,
 1993 happy enters error recovery mode. Depending on whether the grammar file
 1994 declares the two action form `%error { abort } { report }` for
 1995     Resumptive Error Handling,
 1996 it works in one (not resumptive) or two phases (resumptive):
 1997 
 1998  1. Fixup mode:
 1999     Try to see if there is an action for the error token ERROR_TOK. If there
 2000     is, do *not* emit an error and pretend instead that an `error` token was
 2001     inserted.
 2002     When there is no ERROR_TOK action, report an error.
 2003 
 2004     In non-resumptive error handling, calling the single error handler
 2005     (e.g. `happyError`) will throw an exception and abort the parser.
 2006     However, in resumptive error handling we enter *error resumption mode*.
 2007 
 2008  2. Error resumption mode:
 2009     After reporting the error (with `report`), happy will attempt to find
 2010     a good state stack to resume parsing in.
 2011     For each candidate stack, it discards input until one of the candidates
 2012     resumes (i.e. shifts the current input).
 2013     If no candidate resumes before the end of input, resumption failed and
 2014     calls the `abort` function, to much the same effect as in non-resumptive
 2015     error handling.
 2016 
 2017     Candidate stacks are declared by the grammar author using the special
 2018     `catch` terminal and called "catch frames".
 2019     This mechanism is described in detail in Note [happyResume].
 2020 
 2021 The `catch` resumption mechanism (2) is what usually is associated with
 2022 `error` in `bison` or `menhir`. Since `error` is used for the Fixup mechanism
 2023 (1) above, we call the corresponding token `catch`.
 2024 Furthermore, in constrast to `bison`, our implementation of `catch`
 2025 non-deterministically considers multiple catch frames on the stack for
 2026 resumption (See Note [Multiple catch frames]).
 2027 
 2028 Note [happyResume]
 2029 ~~~~~~~~~~~~~~~~~~
 2030 `happyResume` implements the resumption mechanism from Note [Error recovery].
 2031 It is best understood by example. Consider
 2032 
 2033 Exp :: { String }
 2034 Exp : '1'                { "1" }
 2035     | catch              { "catch" }
 2036     | Exp '+' Exp %shift { $1 Happy_Prelude.++ " + " Happy_Prelude.++ $3 } -- %shift: associate 1 + 1 + 1 to the right
 2037     | '(' Exp ')'        { "(" Happy_Prelude.++ $2 Happy_Prelude.++ ")" }
 2038 
 2039 The idea of the use of `catch` here is that upon encountering a parse error
 2040 during expression parsing, we can gracefully degrade using the `catch` rule,
 2041 still producing a partial syntax tree and keep on parsing to find further
 2042 syntax errors.
 2043 
 2044 Let's trace the parser state for input 11+1, which will error out after shifting 1.
 2045 After shifting, we have the following item stack (growing downwards and omitting
 2046 transitive closure items):
 2047 
 2048   State 0: %start_parseExp -> . Exp
 2049   State 5: Exp -> '1' .
 2050 
 2051 (Stack as a list of state numbers: [5,0].)
 2052 As Note [Error recovery] describes, we will first try Fixup mode.
 2053 That fails because no production can shift the `error` token.
 2054 Next we try Error resumption mode. This works as follows:
 2055 
 2056   1. Pop off the item stack until we find an item that can shift the `catch`
 2057      token. (Implemented in `pop_items`.)
 2058        * State 5 cannot shift catch. Pop.
 2059        * State 0 can shift catch, which would transition into
 2060           State 4: Exp -> catch .
 2061      So record the *stack* `[4,0]` after doing the shift transition.
 2062      We call this a *catch frame*, where the top is a *catch state*,
 2063      corresponding to an item in which we just shifted a `catch` token.
 2064      There can be multiple such catch stacks, see Note [Multiple catch frames].
 2065 
 2066   2. Discard tokens from the input until the lookahead can be shifted in one
 2067      of the catch stacks. (Implemented in `discard_input_until_exp` and
 2068      `some_catch_state_shifts`.)
 2069        * We cannot shift the current lookahead '1' in state 4, so we discard
 2070        * We *can* shift the next lookahead '+' in state 4, but only after
 2071          reducing, which pops State 4 and goes to State 3:
 2072            State 3: %start_parseExp -> Exp .
 2073                     Exp -> Exp . '+' Exp
 2074          Here we can shift '+'.
 2075      As you can see, to implement this machinery we need to simulate
 2076      the operation of the LALR automaton, especially reduction
 2077      (`happySimulateReduce`).
 2078 
 2079 Note [Multiple catch frames]
 2080 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~
 2081 For fewer spurious error messages, it can be beneficial to trace multiple catch
 2082 items. Consider
 2083 
 2084 Exp : '1'
 2085     | catch
 2086     | Exp '+' Exp %shift
 2087     | '(' Exp ')'
 2088 
 2089 Let's trace the parser state for input (;+1, which will error out after shifting (.
 2090 After shifting, we have the following item stack (growing downwards):
 2091 
 2092   State 0: %start_parseExp -> . Exp
 2093   State 6: Exp -> '(' . Exp ')'
 2094 
 2095 Upon error, we want to find items in the stack which can shift a catch token.
 2096 Note that both State 0 and State 6 can shift a catch token, transitioning into
 2097   State 4: Exp -> catch .
 2098 Hence we record the catch frames `[4,6,0]` and `[4,0]` for possible resumption.
 2099 
 2100 Which catch frame do we pick for resumption?
 2101 Note that resuming catch frame `[4,0]` will parse as "catch+1", whereas
 2102 resuming the innermost frame `[4,6,0]` corresponds to parsing "(catch+1".
 2103 The latter would keep discarding input until the closing ')' is found.
 2104 So we will discard + and 1, leading to a spurious syntax error at the end of
 2105 input, aborting the parse and never producing a partial syntax tree. Bad!
 2106 
 2107 It is far preferable to resume with catch frame `[4,0]`, where we can resume
 2108 successfully on input +, so that is what we do.
 2109 
 2110 In general, we pick the catch frame for resumption that discards the least
 2111 amount of input for a successful shift, preferring the topmost such catch frame.
 2112 -}

 2211 happyExpectedTokens :: Happy_Int -> Happy_IntList -> [Happy_Prelude.String]
 2212 -- Upon a parse error, we want to suggest tokens that are expected in that
 2213 -- situation. This function computes such tokens.
 2214 -- It works by examining the top of the state stack.
 2215 -- For every token number that does a shift transition, record that token number.
 2216 -- For every token number that does a reduce transition, simulate that reduction
 2217 -- on the state state stack and repeat.
 2218 -- The recorded token numbers are then formatted with 'happyTokenToString' and
 2219 -- returned.
 2220 happyExpectedTokens st sts =
 2221   DEBUG_TRACE("constructing expected tokens.\n")
 2222   Happy_Prelude.map happyTokenToString (search_shifts st sts [])
 2223   where
 2224     search_shifts st sts shifts = Happy_Prelude.foldr (add_action st sts) shifts (distinct_actions st)
 2225     add_action st sts (Happy_GHC_Exts.I# i, Happy_GHC_Exts.I# act) shifts =
 2226       DEBUG_TRACE("found action in state " Happy_Prelude.++ Happy_Prelude.show (Happy_GHC_Exts.I# st) Happy_Prelude.++ ", input " Happy_Prelude.++ Happy_Prelude.show (Happy_GHC_Exts.I# i) Happy_Prelude.++ ", " Happy_Prelude.++ Happy_Prelude.show (happyDecodeAction act) Happy_Prelude.++ "\n")
 2227       case happyDecodeAction act of
 2228         HappyFail     -> shifts
 2229         HappyAccept   -> shifts -- This would always be %eof or error... Not helpful
 2230         HappyShift _  -> Happy_Prelude.insert (Happy_GHC_Exts.I# i) shifts
 2231         HappyReduce r -> case happySimulateReduce r st sts of
 2232           (HappyCons st1 sts1) -> search_shifts st1 sts1 shifts
 2233     distinct_actions st
 2234       -- The (token number, action) pairs of all actions in the given state
 2235       = ((-1), (Happy_GHC_Exts.I# (happyIndexOffAddr happyDefActions st)))
 2236       : [ (i, act) | i <- [begin_i..happy_n_terms], act <- get_act row_off i ]
 2237       where
 2238         row_off = happyIndexOffAddr happyActOffsets st
 2239         begin_i = 2 -- +2: errorTok,catchTok
 2240     get_act off (Happy_GHC_Exts.I# i) -- happyIndexActionTable with cached row offset
 2241       | let off_i = PLUS(off,i)
 2242       , GTE(off_i,0#)
 2243       , EQ(happyIndexOffAddr happyCheck off_i,i)
 2244       = [(Happy_GHC_Exts.I# (happyIndexOffAddr happyTable off_i))]
 2245       | Happy_Prelude.otherwise
 2246       = []
