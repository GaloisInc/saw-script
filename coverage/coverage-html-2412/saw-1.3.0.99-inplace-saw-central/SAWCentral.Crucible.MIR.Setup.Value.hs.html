  170 -- | A enum-related MIR 'SetupValue'.
  171 data MirSetupEnum where
  172   -- | A specific variant of an enum.
  173   MirSetupEnumVariant ::
  174        M.Adt
  175        -- ^ The enum type.
  176     -> M.Variant
  177        -- ^ The variant to use.
  178     -> Int
  179        -- ^ The index of the variant within the list of all the enum's variants.
  180        -- In most circumstances, this index will be the same as the discriminant
  181        -- value, but the index can be different if a variant uses an explicit
  182        -- value. For instance, in this example:
  183        --
  184        -- @
  185        -- enum A {
  186        --     A0,
  187        --     A1,
  188        -- }
  189        --
  190        -- enum B {
  191        --     B0 = 42,
  192        --     B1,
  193        -- }
  194        -- @
  195        --
  196        -- The indexes for @A0@ and @B0@ are both @0@, and the indexes for @A1@
  197        -- and @B1@ are both @1@. The discriminant values are different, however.
  198        -- The discriminants for @A0@ and @A1@ are @0@ and @1@, respectively,
  199        -- while the discriminants for @B0@ and @B1@ are @42@ and @43@,
  200        -- respectively.
  201        --
  202        -- Note that the index is accessible within the 'M.Variant' argument, but
  203        -- retrieving the information is somewhat involved. (See the
  204        -- implementation of @mir_enum_value@.) For this reason, we store this
  205        -- information separately in 'MirSetupEnumVariant' to make it easier to
  206        -- look up later.
  207     -> [MS.SetupValue MIR]
  208        -- ^ The values of the variant's fields.
  209     -> MirSetupEnum
  210 
  211   -- | A symbolic enum value, where the 'M.Adt' represents the enum type.
  212   -- This is only used in the implementation of @mir_fresh_expanded_value@.
  213   -- See @Note [Symbolic enums]@ for a more detailed explanation.
  214   --
  215   -- Note that @repr(transparent)@ enums never use 'MirSetupEnumSymbolic'.
  216   -- Instead, they are represented as a 'MirSetupEnumVariant' where the
  217   -- underlying variant field is symbolic. This makes it simpler to ensure that
  218   -- resolving a @repr(transparent@ enum value will yield a 'MIRVal' whose
  219   -- 'TypeShape' is 'TransparentShape'.
  220   MirSetupEnumSymbolic ::
  221        M.Adt
  222        -- ^ The enum type.
  223     -> MS.SetupValue MIR
  224        -- ^ The symbolic discriminant value.
  225     -> [[MS.SetupValue MIR]]
  226        -- ^ The symbolic values that are used for the fields in each variant.
  227        -- For instance, if one created a symbolic value of this type:
  228        --
  229        -- @
  230        -- enum E {
  231        --     E1(u16),
  232        --     E2(u32, u32),
  233        -- @
  234        --
  235        -- Then the list of fields would be @[[x], [y, z]]@, where @x: u16@

  237        -- @E2@.
  238     -> MirSetupEnum
  239   deriving Show

  269 {-
  270 Note [Symbolic enums]
  271 ~~~~~~~~~~~~~~~~~~~~~
  272 Creating a symbolic MIR enum value is not quite as straightforward as creating
  273 symbolic versions of other compound types, mostly due to the atypical Crucible
  274 representation that enums use. To recap, if we have an enum like this:
  275 
  276   enum E {
  277     E1(u16),
  278     E2(u32, u32),
  279   }
  280 
  281 Then this will be implemented using (roughly) the following Crucible `Type`
  282 under the hood:
  283 
  284   (BVType 64, VariantType [StructType [BVType 16], StructType [BVType 32, BVType 32]])
  285 
  286 Where:
  287 
  288 * The `BVType 64` is a /discriminant/, whose value indicates which variant is
  289   in use. For instance, a discriminant value of `0` indicates that the `E1`
  290   variant is in use, and a discriminant value of `1` indicates that the `E2`
  291   variant is in use.
  292 * The `VariantType ...` indicates that there are two variants in this enum,
  293   where each variant is represented as a struct with the corresponding fields
  294   in that variant.
  295 
  296 At simulation time, VariantTypes are represented as a sequence of
  297 VariantBranches, where a VariantBranch is (roughly) a pair of:
  298 
  299 * A (possibly symbolic) predicate indicating if that variant is being used, and
  300 * A payload representing the fields of the enum. If the predicate does not
  301   hold, then the payload will likely be symbolic, since it does not matter what
  302   the payload value is in that case.
  303 
  304 OK, recap over. Let's get back to the original question: how do we make a
  305 symbolic value of this type? A naïve first attempt is to generate fresh
  306 symbolic values for everything. That is, a symbolic discriminant, as well as a
  307 symbolic predicate and payload for each VariantBranch. While tempting, this
  308 approach won't work. To see why, consider what happens when one pattern matches
  309 on a symbolic enum value. For example, the `match` expression in this function:
  310 
  311   fn foo(x: E) -> u32 {
  312       match x {
  313           E::E1(a) => bar(y),
  314           E::E2(b, c) => baz(b, c),
  315       }
  316   }
  317 
  318 Would turn into the (roughly) following MIR:
  319 
  320   fn foo(x : E) -> u32 {
  321       start_block: {
  322           discr = Discriminant(x, isize);
  323           switchint discr :isize [0, 1] -> [e1_block, e2_block, fallthrough_block]
  324       }
  325       e1_block: {
  326           ... call bar() ...
  327       }
  328       e2_block: {
  329           ... call baz() ...
  330       }
  331       fallthrough_block: {
  332           unreachable;
  333       }
  334       ...
  335   }
  336 
  337 Here, the `switchint discr` statement will check the value of `discr` (the
  338 discriminant), and if it equals `0`, go to `e1_block`; if it equals `1`, go to
  339 `e2_block`; and if it equals something else, go to `fallthrough_block`. In
  340 normal circumstances, `discr` should only ever be equal to `0` or `1`, which
  341 implies that `fallthrough_block` should never be accessible (as indicated by
  342 its `unreachable` instruction).
  343 
  344 Now consider what would happen if the discriminant were an unconstrained,
  345 symbolic value. While a symbolic discriminant could be equal to `0` or `1`, it
  346 could also be equal to any other value! This would spell disaster if Crucible
  347 tried to perform a symbolic branch on, say, `discr == 2`, since that would
  348 cause execution to reach `fallthrough_block` and crash. We want a symbolic
  349 discriminant, but we don't want it to be /that/ symbolic!
  350 
  351 For this reason, after we create a symbolic discriminant value, we also add a
  352 Crucible assumption that the discriminant must be equal to one of the possible
  353 enum variants' discriminants. In the example above, this means that we would
  354 assume assume the following:
  355 
  356   (discr == 0) \/ (discr == 1)
  357 
  358 This way, symbolic execution will never reach `fallthrough_block`. This
  359 Crucible assumption is created in
  360 SAWCentral.Crucible.MIR.Builtins.constructExpandedSetupValue.goEnum`.
  361 
  362 Similarly, we cannot make the VariantBranch predicates completely symbolic, as
  363 whether a predicate holds or not depends on the value of the discriminant. For
  364 this reason, we do not create fresh variables for each predicate, but instead
  365 make each predicate the result of checking the discriminant against particular
  366 values. For instance, the predicate for the `E1` VariantBranch is defined to be
  367 `discr == 0`, and the predicate for the `E2` VariantBranch is defined to be
  368 `discr == 1`. These predicates are defined in
  369 `SAWCentral.Crucible.MIR.ResolveSetupValue.resolveSetupValue`, along with the
  370 fields in the associated payloads.
  371 
  372 Lastly, there are the payloads (i.e., the fields of each variant) in each
  373 VariantBranch. These are created as completely symbolic values—the trick is to
  374 only access the fields when the corresponding predicate holds. For example,
  375 `SAWCentral.Crucible.MIR.Override.matchArg` (in the `MirSetupEnumSymbolic` case)
  376 must be able to match two possibly symbolic enum values together, but it must
  377 be careful to only match the fields in a variant if that VariantBranch's
  378 predicate holds.
  379 
  380 To make this a bit more specific, suppose we have two symbolic enum values
  381 `enumA` and `enumA`, where:
  382 
  383 * `enumA` has the discriminant value `discrA`, and
  384   `enumB` has the discriminant value `discrB`.
  385 * `enumA` has the VariantBranches [(e1_pred_a, [e1_fld1_a]), (e2_pred_a, [e2_fld1_a, e2_fld2_a])], and
  386   `enumB` has the VariantBranches [(e1_pred_b, [e1_fld1_b]), (e2_pred_b, [e2_fld1_b, e2_fld2_b])].
  387 
  388 We only want to match `e1_fld1_a` against `e1_fld1_b` if both enums are using
  389 the `E1` variant, that is, if `e1_pred_a` and `e_pred_b` hold. To this end, the
  390 `matchArg` function checks this by generating (roughly) the following
  391 assertion:
  392 
  393   (discrA == discrB) /\
  394   (e1_pred_a ==> (e1_fld1_a == e1_fld1_b))
  395 
  396 (Note that instead of `e1_pred_a ==> ...`, we could have alternatively used
  397 `e1_pred_b ==> ...`, `(discrA == 0) ==> ...`, or `(discrB == 0) ==> ...`. All
  398 formulations are equivalent.)
  399 
  400 Phew! Enums are surprisingly tricky.
  401 -}
